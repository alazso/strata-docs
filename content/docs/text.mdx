title: "Text"
description: "MiniMessage with PlaceholderAPI and MiniPlaceholders, in the correct resolution order."
---

`TextRenderer` turns MiniMessage strings (with RGB, gradients, and PlaceholderAPI support) into
Adventure `Component`s. It bakes in the one detail that bites everyone:

<Callout type="warn">
  **PlaceholderAPI is resolved *first*, then the result is parsed as MiniMessage.** Doing it the
  other way lets placeholder output be reinterpreted as markup, which is a real class of bug. Strata
  encodes the correct order once so no plugin can reintroduce it.
</Callout>

When PlaceholderAPI isn't installed (or a placeholder misbehaves) the placeholder pass is skipped
gracefully and MiniMessage parsing proceeds unchanged.

```java
TextRenderer text = StrataApi.text();

// Plain MiniMessage (no placeholders):
Component title = text.render("<gradient:#2f8f6f:#0b3d2e>Strata</gradient>");

// Resolve %placeholders% against a viewer, then parse MiniMessage:
Component greeting = text.render("<gray>Welcome, <white>%player_name%", player);

// Many lines at once:
List<Component> lore = text.render(List.of("<red>Line one", "<gray>Line two"), player);
```

## MiniPlaceholders

When [MiniPlaceholders](https://github.com/MiniPlaceholders/MiniPlaceholders) is installed, its
component-safe placeholders are added to every render automatically, so any consumer gets them for
free with no extra setup:

```java
// <luckperms_prefix> and friends resolve as components, not raw strings:
Component name = text.render("<luckperms_prefix><white><player_name></white>", player);
```

Global placeholders resolve everywhere. Audience-scoped placeholders resolve when a target audience
is present in the render context. Unlike PlaceholderAPI's `%...%` placeholders, these are real
MiniMessage tags, so their output is parsed as components and can never be reinterpreted as markup.
When MiniPlaceholders is not installed, the tags are simply left unresolved and parsing proceeds
unchanged.

## Exposing your own placeholders

The other half of the story: define your plugin's placeholders once and expose them through **both**
PlaceholderAPI and MiniPlaceholders, with no per-backend boilerplate. The prefix is your plugin's
name, so a key `balance` is reachable as `%plugin_balance%` and `<plugin_balance>`.

```java
StrataApi.placeholders(this)
    .add("balance", player -> economy.format(economy.balance(player)))   // per-player
    .addGlobal("online", () -> String.valueOf(Bukkit.getOnlinePlayers().size()))
    .register();   // once, in onEnable
```

Each backend is wired only when it is installed, so this is safe whether or not PlaceholderAPI or
MiniPlaceholders is present. A resolver that returns `null` yields no value. Values are inserted as
plain text on the MiniPlaceholders (component) side, so style them where you render, not in the
resolver. Call `unregister()` on disable if you need to tear them down.

## Custom tags

Pass Adventure `TagResolver`s for one-off placeholders without registering anything:

```java
Component msg = text.render(
    "<gray>You earned <reward>!", player,
    Placeholder.parsed("reward", "100 coins"));
```

## Raw placeholder resolution

When you only need the resolved string (for example to compare a placeholder's value), use `resolve`.
It runs PlaceholderAPI without MiniMessage parsing, and returns the input unchanged if PAPI is absent
or the viewer is `null`:

```java
String balance = text.resolve("%vault_eco_balance%", player);
```

This is exactly what the [`papi` condition](/docs/conditions) uses under the hood.
